--------------------------------------------------------------------------------
1. DATASETS (verified links, in order of recommendation)
--------------------------------------------------------------------------------

[1] Railway Track Surface Faults (Mendeley)  -- BEST PRIMARY CHOICE
    URL:  https://data.mendeley.com/datasets/8hxtgyyxrw/2
    Size: 7 defect classes: cracks, squats, spallings, flakings, shellings,
          grooves, joints. Real images from Pakistan Railway, varied
          lighting/weather.
    Why:  Multi-class -> perfect for prototypical networks (N-way K-shot
          episodes). Open download, no access request needed.
    Paper: https://pmc.ncbi.nlm.nih.gov/articles/PMC10828558/

[2] Railway Track Fault Detection (Kaggle, salmaneunus)
    URL:  https://www.kaggle.com/datasets/salmaneunus/railway-track-fault-detection
    Size: 383 images: 191 defective / 192 non-defective.
    Why:  Quick-start binary baseline. Use for the naive-transfer-learning
          ablation.
    Resized 224x224 version:
          https://www.kaggle.com/datasets/gpiosenka/railway-track-fault-detection-resized-224-x-224

[3] Railway Fault -- Rail subset + Fastener subset (Kaggle, ashikadnan)
    Rail:     https://www.kaggle.com/datasets/ashikadnan/railway-track-fault-detection-dataset1-rail
    Fastener: https://www.kaggle.com/datasets/ashikadnan/railway-track-fault-detection-dataset2fastener
    Why:  Fasteners give a SECOND defect domain -- great for the "adapts to
          new defect types" novelty claim.

[4] RSDDs (IEEE DataPort)  -- canonical academic benchmark
    URL:  https://ieee-dataport.org/documents/rsdds
    Size: Type-I: 67 images (160x1000, express rail);
          Type-II: 128 images (55x1250, heavy-haul); pixel-level defect masks.
    Why:  The FS-RSDD few-shot paper you'd compare against uses exactly this.
          Tiny size is BY DESIGN good for few-shot evaluation.
    FS-RSDD paper: https://www.mdpi.com/1424-8220/23/18/7894

[5] Rail-5k (Zenodo)  -- richest taxonomy, but access may be restricted
    URL:  https://zenodo.org/records/4872619
    Size: 5k images, 1.1k annotated, 13 defect types, 0.03mm/px.
    Note: Users report needing to request access -- apply early, don't
          depend on it.
    Paper: https://arxiv.org/abs/2106.14366

[6] SRRSD / SRDA 2024 (Huang et al., Wiley)  -- citation only
    URL:  https://onlinelibrary.wiley.com/doi/10.1111/mice.13087
    Size: 20,662 synthetic + real images.
    Note: NO public download found -- would have to email authors. Treat as
          a citation/comparison, not a data source.

[7] RailSem19  -- not a defect dataset
    URL:  https://openaccess.thecvf.com/content_CVPRW_2019/html/Autonomous_Driving/Zendel_RailSem19_A_Dataset_for_Semantic_Rail_Scene_Understanding_CVPRW_2019_paper.html
    Size: 8,500 ego-view scenes, 20 semantic classes.
    Why:  Useful only for a rail-track SEGMENTATION pre-step to crop the
          rail region before defect detection.

PRACTICAL RECIPE:
  Mendeley (multi-class, episodes) + Kaggle salmaneunus (binary
  baseline/ablation) + RSDDs (benchmark comparison). All three are small
  downloads and openly accessible. Backfill rare classes with augmentation
  or a small diffusion/GAN pass if a class has <20 images.

--------------------------------------------------------------------------------
2. EDGE CASES AND SOLUTIONS
--------------------------------------------------------------------------------

A. DATA & DOMAIN EDGE CASES
---------------------------

1. Normal rail structures mistaken for defects (the #1 false-positive source)
   Problem:  Welds, joints, fasteners, fishplates, and switch points all look
             like "anomalies" to a model trained on clean rail.
   Solution: Include these as explicit HARD-NEGATIVE classes in the support
             set (Mendeley already has "Joints" as a class -- use it). For
             anomaly-detection approaches, make sure the "healthy" memory
             bank includes joints/welds.

2. Rust, dirt, oil stains, water vs. real defects
   Problem:  Surface contamination is visually similar to spalling/corrosion
             but is benign.
   Solution: Heavy color-jitter/grayscale augmentation so the model relies on
             TEXTURE AND GEOMETRY rather than color; add a
             "stained-but-healthy" negative category if you can collect even
             5-10 examples.

3. Specular glare on the railhead
   Problem:  Polished rail reflects sunlight as bright streaks that saturate
             pixels and hide or mimic cracks.
   Solution: CLAHE / histogram equalization in OpenCV preprocessing; augment
             training with synthetic bright streaks; at inference, average
             predictions over multiple frames if you have video.

4. Lighting / weather / time-of-day shift
   Problem:  Model trained on sunny images fails at dusk, in rain, or in
             tunnel shadows.
   Solution: Train-time augmentation (brightness/contrast/shadow overlays via
             albumentations); test-time augmentation; report the few-shot
             curve separately on a "shifted-lighting" split -- this doubles
             as a great demo slide.

5. Motion blur from inspection-vehicle speed
   Problem:  Cracks blur into invisibility above certain speeds.
   Solution: Motion-blur augmentation (directional kernel along track axis);
             a cheap Laplacian-variance blur gate that flags frames as "too
             blurred to assess" instead of silently passing them -- FAILING
             LOUDLY IS A FEATURE for safety systems.

6. Tiny defects in large frames
   Problem:  A hairline crack can be <1% of image area; a whole-image
             classifier averages it away.
   Solution: PATCH-BASED PIPELINE -- segment/crop the rail region (fixed
             geometry, or RailSem19-style segmentation), slice into
             overlapping patches, classify per patch, aggregate. This is also
             exactly how you produce the overlay boxes for the demo.

7. Domain gap: test rail != training rail
   Problem:  Different gauge, rail profile, camera angle, sleeper type
             (Mendeley = Pakistan Railway; RSDDs = Chinese express/heavy-haul).
   Solution: This is the HEADLINE FEW-SHOT STORY: show that 5-10 support
             shots from the NEW domain re-anchor the prototypes, vs. naive
             transfer learning which needs retraining. Cross-dataset
             evaluation (train on one, few-shot adapt to the other) is the
             strongest ablation you can show.

B. FEW-SHOT-METHOD EDGE CASES
-----------------------------

8. Support-set sensitivity
   Problem:  With K=5, one bad/atypical support image wrecks the prototype;
             results vary wildly across random draws.
   Solution: Report MEAN +/- STD OVER >=100 RANDOM EPISODES, never a single
             run (judges who know few-shot will check). Use the class-median
             rather than mean embedding for robustness against one outlier
             shot.

9. Background shortcut learning
   Problem:  The model learns "ballast type A = defective" because the few
             defect shots happen to share a background.
   Solution: Crop tightly to the rail before embedding; verify with Grad-CAM
             that attention is on the rail surface -- a Grad-CAM overlay is
             also a strong demo visual.

10. Unseen/novel defect types at test time
    Problem:  Prototypical networks force every input into a known class.
    Solution: HYBRID DESIGN -- add open-set rejection: if distance to ALL
              prototypes exceeds a threshold, output "unknown anomaly." Or
              pair the few-shot classifier with an unsupervised anomaly
              detector (PatchCore/PaDiM trained on healthy-only data, via
              anomalib: https://github.com/open-edge-platform/anomalib) as
              the first stage. This directly exploits "healthy data is
              abundant."

11. Class imbalance in evaluation
    Problem:  99% of patches are healthy; accuracy and even plain F1 mislead.
    Solution: Report per-class F1 / mAP and a precision-recall curve; pick
              the operating threshold for HIGH RECALL (a missed crack costs
              far more than a false alarm -- say this out loud in the pitch,
              judges love a domain-aware tradeoff).

12. Data leakage between support and query sets
    Problem:  Adjacent video frames or patches from the SAME physical defect
              landing in both train and test inflates metrics massively.
    Solution: Split by DEFECT INSTANCE / CAPTURE SESSION, never by random
              image. The Kaggle sets have near-duplicate frames -- dedupe
              with perceptual hashing (imagehash, ~10 lines) before
              splitting.

13. Pretrained-backbone mismatch
    Problem:  ImageNet features are tuned for objects, not metal-surface
              texture.
    Solution: Use a self-supervised backbone (DINOv2 embeddings work
              remarkably well zero-shot on industrial surfaces) or briefly
              fine-tune the backbone on abundant HEALTHY images with
              SimCLR-style contrastive loss. Cheap, and a clean ablation
              row: ImageNet vs. DINOv2 vs. healthy-pretrained.

C. SYSTEM / DEMO EDGE CASES
---------------------------

14. Threshold calibration across domains
    Problem:  An anomaly score of 0.7 means different things on different
              rail types.
    Solution: Calibrate the threshold per-domain on a small healthy-only
              holdout (you always have healthy data); normalize scores to
              percentiles of the healthy distribution.

15. Camera failure modes
    Problem:  Occlusion (leaves, ballast on rail), defocus, partial rail out
              of frame.
    Solution: A lightweight validity check before inference (is a rail
              detected in frame? is sharpness above threshold?) that emits
              "frame unusable" rather than "no defect" -- distinguish
              ABSENCE OF EVIDENCE from EVIDENCE OF ABSENCE.

16. Live-demo fragility
    Problem:  Real-time video inference dies on stage.
    Solution: Demo on pre-recorded video with live overlay rendering; cache
              embeddings; keep a static-image fallback. Practice the failure
              path.

3. SUGGESTED PIPELINE (mermaid diagram source)
--------------------------------------------------------------------------------

graph LR
    A[Camera / video frames] --> B[Frame validity gate<br/>blur + rail-presence check]
    B --> C[Rail region crop]
    C --> D[Overlapping patches]
    D --> E[DINOv2 / fine-tuned encoder]
    E --> F{Stage 1: Anomaly detector<br/>PatchCore on healthy-only}
    F -- anomalous --> G[Stage 2: Prototypical classifier<br/>5-10 shots per defect class]
    F -- normal --> H[Pass]
    G -- near a prototype --> I[Defect type + box overlay + alert]
    G -- far from all --> J[Unknown anomaly - flag for human review]

KEY DIFFERENTIATOR:
  The two-stage design (healthy-only anomaly screening -> few-shot
  classification with open-set rejection) handles edge cases 10, 11, and 14
  by construction, and maps directly onto the "healthy data abounds, faults
  are rare" framing of the theme.
